chifu splits the work in two. **The CLI detects.** **Your agent fixes.** One rule keeps the loop safe.

## The detect-and-fix loop

<Steps>
  <Step title="Detect">
    `chifu check` reads your manifest and lockfile, resolves the full tree (direct + transitive), and matches every package against **145,000+** CVEs from NVD and OSV. For each vulnerable package it returns the single version that clears all of its CVEs.
  </Step>
  <Step title="Fix">
    Your agent reads the findings and edits files — bumps versions, adds `overrides` / `resolutions` for transitives. Works critical-first. For npm-only automation, `chifu fix` can apply recommended bumps.
  </Step>
  <Step title="Re-check">
    The agent re-runs `chifu check` until `vulnerablePackages` is 0, or only unfixable entries remain.
  </Step>
  <Step title="Hand back">
    You run your own install, build, and tests before committing.
  </Step>
</Steps>

## Why the CLI only detects

chifu does not install or patch anything. No LLM, no secrets. It reads files, queries the vulnerability database, and prints the answer. Small, fast, auditable, safe in CI.

Your agent does the fixing because it already understands your code — changelogs, breaking bumps, call sites.

## The safety rule

While checking and fixing, **never run a full install** — no `npm install`, `npm ci`, `yarn`, `pnpm install`, or `bun install`. Install scripts on compromised packages are the exact supply-chain risk chifu exists to stop.

To re-resolve an npm lockfile after editing `package.json`:

```bash
npm install --package-lock-only --ignore-scripts
```

Downloads nothing, runs no lifecycle scripts. You run a real install only when you're ready.

## Multi-ecosystem

chifu auto-detects one ecosystem per run from your lockfile or manifest. Monorepos with mixed stacks: run `chifu check` from each subproject. See [Ecosystems](/ecosystems).
